# Creative Node Editor (CNE) 软件技术说明手册

## 第一章 软件概述与愿景

### 1.1 项目定位
Creative Node Editor (CNE) 是一款专为数字艺术家、交互设计师及逻辑程序员打造的高性能、模块化节点式编辑环境。它旨在打破传统线性设计工具的束缚，通过直观的图形化节点连接，实现复杂视觉逻辑的快速构建与实时渲染。

### 1.2 核心愿景
在 Web 性能日益强大的今天，CNE 致力于证明“原生即力量”。我们坚持零依赖、Vanilla JS 的技术路线，力求在浏览器内构建出一套媲美原生桌面应用的创意生产工具，为用户提供极致的反馈速度和无限的创意空间。

## 第二章 开发背景与演进过程

CNE 的诞生并非一蹴而就，而是经历了一系列技术突破与架构重构：

### 2.1 概念孵化阶段 (V1.x)
最初的目标是实现一个基础的“点对点”数据流模型。
- **里程碑**：建立了 `Node` 基类与 `Connection` 链路模型。
- **挑战**：早期的渲染直接耦合在节点评估逻辑中，导致在大规模节点场景下出现严重的渲染闪烁与掉帧。

### 2.2 架构重构阶段 (V4.x - V6.x)
为了解决 V1 版本的性能瓶颈，我们引入了 **V2 渲染流水线**。
- **核心变更**：实现了解耦的渲染池（RenderPool）系统。
- **突破**：引入了 `TransformNode`，支持基于数学表达式的动态几何变换，软件开始具备处理复杂动画的能力。

### 2.3 交互深化阶段 (V7.x)
重点转向用户体验与工程化支持。
- **功能引入**：增加了快捷键系统（Ctrl+C/V/Z）、视口地图（Minimap）以及完善的右键上下文菜单。
- **底层支持**：实现了本地文件系统关联，允许用户直接在浏览器中操作本地硬盘图片素材。

### 2.4 性能优化与架构完善 (V8.0++)
这是 CNE 走向成熟的关键节点。
- **批量处理**：实现了多选模式下的拓扑感知复制粘贴（Smart Clipboard），解决了复杂逻辑组的快速复用问题。
- **稳定性增强**：针对 `ImageNode` 进行了多轮重构，解决了远程跨域、内存溢出及加载死循环等核心痛点，最终确立了“本地化资产优先”的原则。

## 第三章 软件系统架构深度解析

CNE 的卓越性能源于其高度系统化的分层架构：

### 3.1 核心引擎层 (Core Logic)
- **NodeCanvas (交互中枢)**：管理无限缩放画布，处理复杂的鼠标/触摸事件流，实现高精度的对象拾取与选择。
- **ConnectionManager (拓扑专家)**：维护 DAG（有向无环图）结构。每次连线变更时，它都会自动执行拓扑排序，计算出最优评估序列。
- **RenderEngine (执行核心)**：负责节点的实时评估（Evaluation）。它通过监听数据脏标记（Dirty Flags），仅对必要节点进行重计，极大地节省了 CPU 算力。

### 3.2 级联渲染管线 (V2 Pipeline)
CNE 采用了一种创新的三级渲染模式：
1. **指令生成 (Node Level)**：每个节点根据其内部逻辑生成一组 Canvas 绘图指令。
2. **缓冲汇聚 (RenderPool Level)**：专门的 RenderPool 接收所有上游指令，按照深度（Z-Index）进行分层排序与冲突处理。
3. **最终输出 (Screen Level)**：Renderer 将汇聚后的指令一次性渲染至离屏 Canvas，再通过主 Canvas 呈现，彻底解决了画面撕裂问题。

### 3.3 UI 响应系统层 (UI Layer)
- **ParamPanel (智能面板)**：这是一个自适应的 UI 生成器。它能读取任何节点的参数定义（Params Definition），动态创建出包含滑块、取色器、表达式输入框的控制矩阵。
- **事件闭环**：UI 控件的调整会触发节点的 `markDirty()`，随后 RenderEngine 立即重绘，形成了一个低于 16ms 的完整闭环。

## 第四章 核心技术环节与创新

### 4.1 拓扑感知的智能剪贴板 (Smart Clipboard)
这是 CNE 的技术护城河之一。传统的节点编辑器在复制多个节点时往往会丢失它们之间的连线。
- **技术细节**：在 `copyNodes` 阶段，系统不仅序列化节点属性，还会扫描并标记选中节点内部的所有 `Internal Connection`。
- **复位逻辑**：粘贴时，系统会生成一组临时的 ID 映射表，通过重建算法在瞬间让 A' 与 B' 恢复它们在原主 A 与 B 处的关系。

### 4.2 动态表达式评估引擎 (Expression Engine)
我们在 `TransformNode` 等节点中嵌入了动态 JS 环境。
- **核心逻辑**：利用 JavaScript 的动态闭包特性，将用户的输入字符串编译为高效的匿名函数。
- **参数注入**：在每一帧渲染前，系统将时间、鼠标位置等全局变量注入闭包，实现了“代码即动画”的效果。

### 4.3 高效的资产管理与内存复用 (Memory Strategy)
针对大尺寸本地图片：
- **Blob 映射**：放弃了效率低下的 Base64 编码，全面采用内存 Blob 映射（Object URL）。
- **生命周期管控**：严谨的 `URL.revokeObjectURL` 逻辑，确保切换上千张素材也不会导致浏览器标签页崩溃。

## 第五章 软件主要功能概览

### 5.1 渲染类节点 (Renderable Nodes)
- **Image Node**：支持多格式（PNG/JPG/WebP/DataURL）的本地图片加载与实时变换。
- **Text Node**：提供丰富的排版参数，包括字体、行间距、描边效果等。
- **Canvas Node**：更底层的手绘指令节点。

### 5.2 变换类节点 (Transformation Nodes)
- **Transform**：提供平移 (Translate)、旋转 (Rotate)、缩放 (Scale) 的级联控制。支持中心点偏移设置。

### 5.3 逻辑类节点 (Logic Nodes)
- **Container**：实现数据的批量分发与数组映射。
- **String/Number Param**：全局参数控制器，用于集中管理多个节点的共有属性。

## 第六章 软件特点分析

1. **极致原生 (Ultra-Vanilla)**：不依赖 React/Vue/Three.js，实现了 0KB 的第三方运行时负担。
2. **所见即所得 (WYSIWYG)**：任何逻辑连线的改动都能在 1/60 秒内反映在画布上。
3. **高扩展性**：基于 ES6 类继承的架构，新增一种节点类型通常只需不到 50 行代码。
4. **演示级设计**：内置全屏预览（F11 兼容），特别优化的“纯净模式”适合现场新媒体演出。

## 第七章 软件使用细则

### 7.1 环境准备
建议在最新版本的 Google Chrome 或 Microsoft Edge 浏览器中运行。
- **开发地址**：本地运行 `npm run dev` 后访问 `http://localhost:5173`。

### 7.2 画布操作
- **平移浏览器**：使用鼠标中键或右键在空白处点击并拖拽。
- **缩放**：使用鼠标滚轮，缩放将以光标为中心。
- **多选**：在空白处按住左键并拖动出框选框，或者按住 `Shift` 加点击进行连选。

### 7.3 节点维护
- **连线**：点击输出端口并拖出，吸附至输入端口。
- **断开**：点击已占有的输入端口即可移除该条连线。
- **参数解耦**：点击属性标签左侧的圆形标记（端口开关），可以手动控制该属性是否接受外部连线。

## 第九章 有待处理的问题与挑战

### 9.1 图片资产的自动重载限制 [IMPORTANT]
由于 Web 浏览器极高的安全沙箱限制，JavaScript 无法直接获取文件的绝对物理路径。
- **现象**：当用户保存工程（.cne）后重新加载，虽然节点保存了图片名，但浏览器无法自动从本地磁盘读取数据。
- **当前对策**：用户在加载工程后，需要点击图片节点的“文件夹”图标进行一次手动关联。
- **技术瓶颈**：这是一个浏览器安全范式的硬限制，除非借助本地转发服务器或未来成熟的 File System Access API。

### 9.2 复杂路径下的撤销/重做
目前系统支持简单的删除撤销，但对于大规模连线变更的 Undo 堆栈及其对拓扑排序的一致性影响，仍需进一步优化以防止逻辑死锁。

## 第十章 优化计划与未来展望

### 10.1 WebGPU 渲染核心升级
计划将主渲染器从 Canvas 2D 升级至 WebGPU。这将使得处理 100,000 以上的活跃节点成为可能，支持 3D 网格渲染与更高级的着色器特效。

### 10.2 逻辑封装 (Nodes Grouping)
开发“子图”功能。用户可以将一整套复杂的逻辑回路（如自制的粒子发生器）封装进一个单一的盒子节点中，从而实现逻辑的高度复用。

### 10.3 跨平台离线版 (Electron)
借助 Electron 技术栈，将 CNE 打包为桌面应用，从而突破浏览器的文件访问限制，解决图片自动加载的痛点，并提供更强的本系统资源调用能力。
